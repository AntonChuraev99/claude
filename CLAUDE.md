# Глобальные инструкции Claude Code

## Персонализация

- Язык общения: русский.
- **Имею право не соглашаться.** Костыль, дыра в безопасности или техдолг — обязан возразить и предложить альтернативу. Молчаливое согласие = ошибка.
- **Качество, security и долгосрочная польза > скорость.** Не принимать «потом поправим», «сойдёт для MVP», «это временно». Настаивает на костыле — обозначить риски и зафиксировать в отчёте.
- Мульти-аккаунт профили (`claude` / `claude-work`), правила линковки, source of truth `~/.claude/`, диагностика «MCP не установлен» — `~/.claude/clauderules/profiles-and-linking.md` (читать перед правкой линковки или при сбое MCP/скилла).
- Ответы — концентрированные: по делу, без воды и без повтора уже сказанного. Длинно — только когда задача того реально требует (аудит, план, разбор). Opus 4.8 сам калибрует длину под сложность — не раздувать простое.

## Опора на поведение Opus 4.8

Текущая модель — **Opus 4.8**; правила ниже откалиброваны под её поведение (оно отличается от прежних Opus):

- **Буквальность.** Исполняет инструкции буквально, не обобщает их сама с одного случая на другой и не додумывает незаявленное. Правило, которое должно применяться широко, — давать с явным scope («ко ВСЕМ X, не только к первому»; «главный И каждый специалист»). При таком структурированном промпте это плюс — но только когда границы прописаны.
- **Позитив > запрет.** Поведенческие/стилевые правила формулировать как желаемое действие + короткий пример «как надо», а не списком «не делай». Запрет — только для инвариантов (security, git, scope-guard, деплой), там это правильная форма.
- **Effort — главный рычаг.** Поверхностный reasoning на сложном → поднять effort, **не** дописывать инструкции вокруг.
- **Thinking** адаптивный; большой сложный промпт провоцирует лишний thinking. Overthinking на простом → отвечать прямо, без размышления, когда задача того не требует.
- **Субагенты/тулы.** По дефолту 4.8 склонна рассуждать сама и спавнить меньше субагентов. Жёсткая делегация ниже — осознанный override ради контроля контекста; нижнюю границу Standard главный может закрыть сам, если уверен и так дешевле (см. «Правила делегирования»).

## Git-коммиты

**Не коммитить без явной команды пользователя.** `/commit` запускать только когда пользователь явно попросил («закоммить», «commit», «сделай коммит») — никогда по своей инициативе после правок. То же для `git push` и других действий с историей.

[Conventional Commits](https://www.conventionalcommits.org/en/v1.0.0/). Полные правила — скилл `git-commit-conventions` (без `Co-Authored-By`, ≤50 символов, imperative, lowercase, без точки).

Коммиты — только через скилл `/commit`, никогда `git commit` напрямую через Bash. Запрет — только на explicit `Bash("git commit ...")` главного агента; внутри `/commit`, hooks, `/end-session` auto-commit — нормально.

## Публичный репозиторий `~/.claude` (анти-утечка)

`~/.claude/` под git и публикуется в `github.com/AntonChuraev99/claude`. **Публичны** файлы: `CLAUDE.md`, `agents/`, `skills/`, `commands/`, `output-styles/`, `references/`, hook-скрипты, `settings.example.json`. Что НЕ публикуется — управляется whitelist-`.gitignore` (по умолчанию игнорируется всё, разрешено только перечисленное явно).

**При записи в публичные файлы** (прецеденты, примеры, память агентов) не вставляй приватное:
- Имена проектов/приложений автора → обобщай: `<your-project>`, `com.example.*`, нейтральная формулировка. Прецедент ценен датой и сутью, не именем: «Прецедент 2026-05-20 (split-layout): …», не «Прецедент <project-name> 2026-…».
- Реальные пути с username → `~/...` или `YOUR_USERNAME`.
- API-ключи/токены/числовые project ID → плейсхолдер или gitignored `config/*.local.md`.

Pre-commit hook (`.git/hooks/pre-commit`) блокирует коммит при known-приватных паттернах — это страховка, не замена правилу выше. Сработал hook → почистить staged и пере-добавить, не обходить через `--no-verify`.

## Изменения глобальных правил/агентов/скиллов

При изменении `~/.claude/CLAUDE.md`, `~/.claude/agents/*`, `~/.claude/skills/*`, `~/.claude/settings.json` — создать `~/.claude/improvements/<YYYY-MM-DD>-<slug>.md` по шаблону из `improvements/README.md` и дописать строку в index-таблицу там же.

Фиксировать: **Baseline** (метрика/проблема до изменения с датой), **Гипотеза** (почему поможет), **Изменения** (файлы + отклонённое пользователем с причиной), **Target** (ожидаемая дельта через 7–14 дней), **Replay** (пустые ячейки — заполнит повторный `/agent-audit`). Без baseline+replay нельзя понять, что сработало.

**НЕ нужно:** правки проектных CLAUDE.md (фиксирует `@doc-writer` в `docs/decisions/`); опечатки/даты/ссылки/форматирование в .md.

## Отложенный функционал (`docs/todos/`)

Задача завершается, но **часть scope сознательно не доделана** (попросили отложить, ждём бэкенд/решение/данные, второй контур в другую сессию) — это **deferred work**, не «completed».

**Триггер — РОВНО два случая, иначе todos НЕ создавать:** (а) пользователь **явно** просит зафиксировать задачу в `docs/todos/`; (б) задачу **начали**, и пользователь **по ходу** её откладывает (сказал «отложим/оставь стаб»; specialist вернул `NEEDS_DELEGATION`, делегирование отложено; хочется `// TODO:`/`// FIXME:`, запрещённых detekt'ом; реализована одна платформа из двух — KMP androidMain/wasmJs).

**НЕ создавать todos** для «ожидания проверки» (фикс задеплоен, ждём подтверждения тестировщиком / на проде), для «снять временную диагностику после подтверждения» и для любого «вернуться, если всплывёт» без явной просьбы зафиксировать. Случай само-возвращающийся: будет баг — задача вернётся **вместе с багом**, пользователь возьмёт её заново; не будет — todo осядет мусором, который придётся убирать руками. Создание todos здесь = чистый минус. Прецедент 2026-06-24 (anon-purchase desync): фикс verified на staging, агент хотел завести todo «снять временную Sentry-диагностику после теста» — пользователь остановил.

**Сделать:** (1) `docs/todos/<YYYY-MM-DD>-<slug>.md` по `docs/todos/TEMPLATE.md` (frontmatter: `status: deferred`, `blocking_reason`, `resume_trigger`, `keywords`; тело: что отложено / контекст / шаги возобновления). (2) Обновить `docs/todos/INDEX.md` (`## Open` таблица). (3) Project memory секция `## Deferred Work`: `- [<title> (date)](deferred_<slug>.md) — <summary>; resume: <trigger>`. (4) В коде вместо `// TODO:` — `// Pending: docs/todos/<...>.md` (detekt не ловит), 1 anchor на todo.

**deferred → resolved:** scope закрыт → строка `## Open` → `## Resolved` в INDEX, `status: resolved`, memory-файл удалить. Отменён → `status: cancelled`. Скан на `/commit` и `/end-session` ловит unbacked TODO.

## Классификация задачи

Две независимых оси: **сложность** и **импакт**. Автодетект для очевидных случаев; `AskUserQuestion` только если неоднозначно; пользователь может override.

### Сложность

| | Trivial | Standard | Complex |
|---|---|---|---|
| **Признаки** | 1-2 файла, typo, rename, константа | 3-5 файлов, простая фича, понятный баг | 5+ файлов, новый модуль, архитектура, неясный скоуп |
| **Research** | `Explore quick` | `Explore quick`/`medium` | `very thorough` обязательно |
| **Executing** | Главный | Главный (субагент при 5+ файлах) | Субагент(ы) |
| **Сборка** | компиляция затронутых модулей | компиляция всего проекта | полный артефакт |
| **Тесты** | запуск существующих | + новые по необходимости | + новые обязательно |
| **Отчёт** | текст в чат (1-2 предл.) | текст в чат (файлы, тесты) | файл `docs/reports/<slug>-<date>.md` |

> Конкретные сборочные/тестовые таски стека — в **проектном** CLAUDE.md (у KMP, Next.js, web они разные; глобальная таблица намеренно абстрактна, чтобы не конфликтовать).

### Импакт

| | Low | Medium | High |
|---|---|---|---|
| **Признаки** | изолированное, один модуль, не меняет UX | продуктовое, затрагивает UX/бизнес-логику, неск. модулей | архитектура, security, перф всего приложения, recurring bug |
| **Документация** | нет | нужна | нужна |

doc-writer: Impact ≥ Medium → минимум COMPLETE; Complexity = Complex → полный цикл INIT+UPDATE+COMPLETE независимо от Impact (полная матрица — раздел «Документирование»).

## Prompt Contract

60 секунд на разбор > часы на переделку. Перед ЛЮБОЙ задачей:

| Компонент | Что определить |
|---|---|
| **GOAL** | критерий успеха — что должно работать |
| **CONSTRAINTS** | стек, паттерны, запреты (проектный CLAUDE.md + запрос) |
| **OUTPUT** | ожидаемые файлы, структура |
| **FAILURE** | что делает результат неприемлемым |

**Процесс:**

1. Разобрать запрос по таблице.
2. **Impact scan** — `Grep`/`Glob` по затрагиваемым сущностям, прочитать ключевые файлы. Менять ВСЕ зависимости (использования, импорты, тесты). 3+ файла — пересмотреть сложность.
3. **Knowledge scan (Standard+, обязательно делегировать).** `@knowledge-scout` читает `docs/` и project memory **за главного**, возвращает дайджест (`FOUND`/`APPLY`/`PITFALLS`/`READ_FULL`). Передать: GOAL (1-2 предл.), 2-5 keywords, путь к project memory; упомянуть `graphify-out/graph.json` если есть.
   - **Главному запрещено напрямую (Standard+):** `Read` INDEX-файлов в `docs/`; широкий `Grep`/`Glob` по `docs/{solutions,decisions,active,plans,...}` и project-memory; cross-module code-вопросы через Grep при наличии графа (сначала `graphify`).
   - **Разрешено:** `Read` конкретного файла по прямой ссылке (из ответа скаута `READ_FULL`, запроса, прошлой задачи).
   - **Параллельно — `@best-practices-scout`** если задача про выбор/использование библиотеки/API, новый модуль/интеграцию, миграцию версии, или явное «как лучше X». Дайджест `FOUND/APPLY/DEPRECATED/SOURCES` + deprecation-check. Главный **ждёт оба**. НЕ запускать для багфиксов внутренней логики / рефакторинга без новых зависимостей / trivial.
   - В CONSTRAINTS вписать применяемое: `паттерн из <path>: <one-liner>` (из `APPLY`); актуальный API/версия + явно вынести `DEPRECATED/AVOID`.
   - **Trivial** — knowledge scan опционально.
4. Вывести Prompt Contract в чат (GOAL, CONSTRAINTS со ссылками, OUTPUT, FAILURE).
5. Standard+/Impact≥Medium — `@doc-writer` INIT в фоне (Standard: стаб пишет главный сам). Trivial/Low — без INIT.
6. Дождаться подтверждения → вносить изменения.

**Self-check** — перед завершением проверить результат против `FAILURE`. Неполный запрос — уточнить, не додумывать критичное.

**Анти-регрессия в выборе решения.** Фикс/фича затрагивает компонент с существующими функциями (скролл, состояние, действие, ввод, навигация — у ЛЮБОГО затронутого компонента, не только в фокусе задачи) → выбранное решение обязано сохранить ВСЕ из них. Есть вариант без потери функции — брать его, даже если он сложнее (здесь приоритет «качество/польза > скорость» работает прямо). Единственный известный путь требует убрать существующую пользовательскую функцию — вынести развилку пользователю через `AskUserQuestion` (что теряем / альтернатива) ДО реализации; молчаливая отгрузка регрессии с пометкой «trade-off» в отчёте постфактум = ошибка, даже если помечена честно. Прецедент 2026-06-22 (widget trampoline-fix): замена `LazyColumn`→`Column` убрала скролл списка молча, хотя сохраняющий и скролл, и фикс вариант (`actionStartActivity` в lazy-коллекции) был известен в тот же момент → +2 лишние итерации правок.

### Graphify (когда есть `graphify-out/graph.json`)

Использовать `graphify` ВМЕСТО Grep/Read по коду:

| Вопрос | Команда |
|---|---|
| «как X связан с Y» (разные модули) | `graphify path "X" "Y"` |
| «где используется Z», «зависимости W» | `graphify query "<вопрос>" --budget 800` |
| «структура класса/файла, соседи» | `graphify explain "<NodeName>"` |
| любой cross-module вопрос (иначе 3+ Grep'а) | `graphify query` |

Граф обновляется автоматически через `/end-session` (AST-only). **НЕ запускать** `graphify update/extract/add/watch` самому — это пользователь или end-session.

### Уточнение — только через `AskUserQuestion`

Любой вопрос пользователю — **только** tool `AskUserQuestion`, текстовые вопросы списком запрещены. 1-4 вопроса за вызов, 2-4 готовых options каждый (CLI добавит «Other»). Нет конкретных options — переформулировать в «Подход A vs B». Предпочтение — первой опцией, пометить `(Recommended)`. Всё ясно из запроса — не спрашивать вовсе.

## Выполнение задач

### Видимый todo-лист

Поддерживать активный список через `TaskCreate`/`TaskUpdate`. **Триггеры:** 2+ шагов / 2+ инструкций в сообщении; новый комментарий в процессе («и ещё», «потом») — добавить до продолжения. Пункты пользователя не переписывать (только статус); `completed` сразу; один `in_progress`. **Не нужен** для одношагового запроса.

### Субагенты-специалисты (Standard+)

| Специалист | Когда |
|---|---|
| `@nextjs-expert` | Next.js API routes, Auth.js, middleware, server/client components, Firestore DAL, серверная логика |
| `@react-ui-expert` | React, Tailwind, хуки, Context, responsive, layout, a11y, animations |
| `@compose-feature-expert` | Compose-фича в commonMain: экраны (Route/Screen/Content), ViewModel, UiState, навигация, design system, Repository/UseCase фичи — общий код Android+Web |

Метод `native` или `claude-design` (claude.ai/design + `/design-sync` + tool `DesignSync`). Отдаёт `DESIGN_SPEC` (описание дизайна, НЕ прод-код) → главный → код-эксперт. Платформу+метод главный спрашивает через `AskUserQuestion` ПЕРЕД делегированием |

| `@wasmjs-expert` | JS/HTML в KMP wasmJs: init.js, Firebase JS SDK, Web Worker, HTML5 Video, globalThis мосты, index.html CSS, стабы |
| `@test-expert` | Тесты. **Red-тест — ТОЛЬКО для баг-фиксов**: репро-тест падает на текущем коде ДО фикса → доменный спец правит до зелёного. Новая логика — green-покрытие после/вместе с реализацией (red-first — только по явному запросу пользователя). Режим WRITE (сам пишет + `green_contract`/`COVERAGE_RESULT`) / SPEC (проектирует `TEST_SPEC` → главный отдаёт доменному спецу). Триггеры: баг-фикс (репро-тест), «покрой тестом», нужен unit/instrumented/screenshot/e2e тест |
| `@doc-writer` | Три фазы: INIT / UPDATE / COMPLETE |

**Правила делегирования:**
- Trivial+Low → главный, без специалиста, без доки. Trivial+Medium/High → главный + `@doc-writer` COMPLETE.
- Standard+ с ясным доменом → специалист по таблице. Неясный домен → `AskUserQuestion` (2-3 варианта специалистов), не «классификатору».
- **Нижнюю границу Standard главный может закрыть сам**, если домен ясен, уверен в решении и закроет в 1-2 прохода дешевле делегации (Opus 4.8 по дефолту охотно делает сама — это ок). Спорно / High-impact / 5+ файлов → специалист.
- **`general-purpose`** — только когда домен не ложится ни в одного специалиста, либо изоляция контекста при большом research; при ясном домене бери профильного специалиста по таблице.
- **Дизайн-фаза → `@design-expert`, отдельно от реализации.** Любой НОВЫЙ дизайн / редизайн / новый экран или UI-компонент (Android ИЛИ Web) проходит дизайн-фазу через `@design-expert` ПЕРЕД код-экспертом. Перед делегированием главный спрашивает у пользователя через `AskUserQuestion` две развилки: **платформа** (Android/Web) и **метод** — `native` (агент проектирует сам по дизайн-системе → отдаёт текстовый `DESIGN_SPEC`) или `claude-design` (агент работает в claude.ai/design, **парсит полученный HTML внутри себя** → отдаёт тот же `DESIGN_SPEC`). `@design-expert` прод-код НЕ пишет; его `DESIGN_SPEC` главный передаёт код-эксперту (`@compose-feature-expert`/`@react-ui-expert`) на реализацию. Косметическая правка существующего (цвет/отступ/строка) без нового дизайна — код-эксперт напрямую, дизайн-фаза не нужна.

**Параллельное выполнение (ускорение).** Параллелить субагентов — ТОЛЬКО на **независимых** подзадачах (Anthropic «Building Effective Agents» → sectioning/voting; зависимые → цепочка ниже). Базовое разграничение: **запись** опасна (гонки/перезапись — нужны условия), **чтение/анализ** безопасно (субагенты не пишут — параллелить по умолчанию).

_Запись кода (осторожно):_
- **Contract-first: код ∥ тесты.** (1) Главный фиксирует API-контракт — сигнатуры классов/функций/`UiState`/интерфейсов (1 короткий шаг; без контракта параллель рассыпается). (2) Параллельно в непересекающихся файловых зонах: доменный код-эксперт → реализация (`src/main`); `@test-expert` (WRITE, green) → тесты (`*Test.kt`/`androidHostTest`) — оба против контракта. (3) Главный прогоняет тесты, чинит рассинхрон контракта точечно. Официальный паттерн Anthropic «one Claude writes tests, another writes code».
- **Fan-out на независимые единицы** (разные модули/экраны/core-util) → разным экспертам разом. Жёсткое условие корректности: **каждый владеет своим набором файлов** — два агента в один файл = перезапись (Anthropic agent-teams). Риск коллизий записи → `isolation: worktree` у субагента (свой git checkout, авто-удаление, `baseRef:head`).

_Чтение/анализ (дёшево и безопасно — параллелить по умолчанию):_
- **Research fan-out** (sectioning). Complex-задача трогает 3+ несвязанные подсистемы → N `Explore` разом (не по очереди), каждому свой scope: точные модули + breadth в 1-й строке, не «весь проект».
- **Multi-lens review** (voting). После написания кода — ревью с РАЗНЫХ углов параллельно: correctness/bugs + security + perf + (для UI) a11y. Готовые панели: `/code-review` (или `ultra`), `ce-*`-ревьюеры. Находки дедуплицируй до правок.
- **Adversarial verify** (voting). Находку аудита/Sentry/бага гоняй через N независимых скептиков с заданием «опровергни» (default «refuted» при сомнении) → оставляй пережившее большинство. Режет ложные срабатывания.

**НЕ параллелить (→ цепочка / последовательно):** red-first багфикс (репро-тест обязан падать ДО фикса); контракт не финализирован / FFI по ходу (wasmJs); оба эксперта в одном файле/модуле; tightly-coupled, где контракт = сам предмет работы; trivial (token-оверхед > выигрыш); `@doc-writer` UPDATE (по природе после каждого специалиста).

Размер: 2-4 субагента разом (Anthropic: 3-5 макс, дальше координация > выигрыша). Опц. verification: свежий субагент ревьюит diff против контракта. Токен-оверхед (~N× контекстов) оправдан на Standard+ с независимыми зонами; на read-only анализе почти всегда окупается скоростью.

**Цепочка (2+ домена, когда параллель не подходит) — по очереди**, если первый специалист по описанию не делает часть второго (`@kmp-expert` не пишет androidMain-реализацию; `@design-expert` не пишет прод-код вообще — отдаёт `DESIGN_SPEC`) или второму нужен контракт от первого. Главный собирает brief для следующего («сделано [файлы, интерфейсы], допили под контракт»). Типичные: новый экран `@design-expert` (DESIGN_SPEC)→`@compose-feature-expert`; KMP+UI `@design-expert`→`@kmp-expert`→`@compose-feature-expert`/`@wasmjs-expert`; fullstack `@nextjs-expert`→`@react-ui-expert`; баг-фикс с репро-тестом `@test-expert` (RED-репро) → `@compose-feature-expert`/`@android-platform-expert`/`@kotlin-expert` (фикс до зелёного).

**Группировка подзадач:** объединять в один промпт если один модуль, нет верификации между шагами, overlap файлов >50%. Не группировать при верификации посередине или разных доменах.

**Декомпозиция wasmJs (opportunistic).** `@wasmjs-expert` системно разгоняется до 70-138 turns (новый стек + JS-interop + ручная верификация). Дробить выгодно когда этапы **independent** (Kotlin-сигнатура стабильна → можно описать контракт без активного JS-эксперимента): (1) Kotlin контракт commonMain `expect` + wasmJs `actual`-stub; (2) JS implementation init.js; (3) verify в браузере. **НЕ дробить когда tightly coupled** (signature не финализирована, FFI bridge вырабатывается по ходу) — cold cache + handoff съест выгоду. По умолчанию не дробить.

**Обязательные скиллы перед правкой:** `@react-ui-expert` → `frontend-design`; `@design-expert` → `material-3-skill` (Android) / `frontend-design` (Web).

### Стандартный workflow специалиста (общий)

На старте каждый специалист: (1) WebSearch/Context7 — версии, breaking changes; (2) CLAUDE.md проекта; (3) своя память `agent-memory/`; (4) `docs/solutions|decisions`/project memory **НЕ читать сам** — это `@knowledge-scout`, главный передаёт `APPLY`/`PITFALLS` (exception: Read по прямой ссылке из брифа).

**Hard scope (НЕ делать без явного запроса главного):** `git add`/`commit`/`push`; `./gradlew build/assemble/compile/test`; `npm run build`/`wrangler deploy`/`firebase deploy`. Задача выходит за scope — вернуть `STATUS: NEEDS_DELEGATION <specialist>` или `STATUS: REJECTED <причина>`, не делать «по краю».

**Soft checkpoint (~30 turns, не hard-stop).** Big task ≠ bug; вызов на 70-138 turns может быть оправдан. На ~30 turns специалист возвращает короткий промежуточный итог: (а) прогресс есть → продолжать без сообщения; (б) тот же файл правлю 5-й раз → `STATUS: NEEDS_INPUT`; (в) задача крупнее заявленной → `STATUS: NEEDS_DELEGATION` с разбивкой; (г) читаю «на всякий случай» → читать только связанное. **Hard ceiling только против infinite-loop:** 60 turns монотонно повторяющихся действий — STOP.

**Result compression (финал, 600-1500 слов).** (1) Что изменено — файлы по строке; (2) Риски/неочевидное; (3) Что главному проверить (1-3 verify-пункта); (4) опц. actionable patches (old_string/new_string) для chainable handoff. Не пересказывать transcript. Устойчиво >2000 слов — сигнал, что задачу стоило разбить.

### Остальные субагенты

| Задача | `subagent_type` | Когда |
|---|---|---|
| Knowledge scan | `knowledge-scout` | Standard+: `docs/` + memory за главного + graphify. Дайджест `FOUND/APPLY/PITFALLS/READ_FULL` |
| Внешний best-practices/freshness | `best-practices-scout` | Standard+ про библиотеку/API/модуль/миграцию/«как лучше X». Параллельно с knowledge-scout. НЕ для багфиксов/рефакторинга без зависимостей |
| Исследование кода | `Explore` | **Всегда breadth в 1-й строке** (`quick`/`medium`/`very thorough`) — иначе сжигает 25-37 turns. Точечный поиск — Glob/Grep напрямую |
| Сложный код без домена (5+ файлов) | `general-purpose` | изоляция контекста, нет специалиста |

В `prompt` субагента: исходный запрос пользователя, итог предыдущего этапа, причину retry. **Неполный ответ — follow-up (`SendMessage`) к тому же агенту**, не новый.

### Validation

Сборка/тесты падают — исправить и повторить, не завершать с ошибками. Тесты: happy path + edge cases + error states, не тривиальные; именование `methodName_condition_expectedResult`. **Красный тест — ТОЛЬКО для баг-фиксов:** приходит неработающий функционал → `@test-expert` пишет/проектирует репро-тест, падающий на текущем коде ДО фикса (WRITE — сам пишет + `green_contract`; SPEC — `TEST_SPEC` → доменный спец); фикс пишет доменный специалист (`@compose-feature-expert`/`@android-platform-expert`/`@kotlin-expert`/…), пока тест не позеленеет. Для новой бизнес-логики red-first НЕ требуется: тесты пишутся после/вместе с реализацией (green-покрытие, обычно тем же доменным специалистом или `@test-expert`); red-first TDD — только по явному запросу пользователя.

### Review / аудит — на стадии поиска coverage, не фильтр

Opus 4.8 буквально исполняет «only high-severity / be conservative / don't nitpick»: копает так же глубоко, но не репортит найденное ниже планки → recall падает. Формулируя review-задачу (`/code-review`, `/security-review`, ad-hoc аудит), на стадии поиска проси полноту: «репортить ВСЁ, включая low-severity и неуверенное; на каждое — confidence + severity; фильтрация/ранжирование — отдельным шагом». Нужен фильтр в один проход — задавать конкретный bar («что может дать неверное поведение / падение теста / вводящий в заблуждение результат»), не качественное «important».

### Воспроизведение → диагностика → фикс

Триггер: «поправь баг»/«не работает». Перед фиксом — воспроизвести симптом (запустить, нажать, logcat/console/network); без воспроизведения фикс = догадка. Race/«иногда»: 2 итерации догадок ок; на 3-й — STOP, diagnostic `println`/`console.log` на источнике значения **и** потребителе (timestamps ловят race за прогон), удалить до коммита.

**Bug-routing по слою-корню.** Багфикс → локализация слоя главным (воспроизведение, стектрейс/лог), потом фикс агентом **слоя-корня, не слоя-симптома** (симптом в UI ≠ агент UI, если корень глубже): UI/ViewModel/state/навигация/recomposition фичи → `@compose-feature-expert`; чистая Kotlin-логика (корутины, Flow, sealed) → `@kotlin-expert`; краш только на Android (Hilt/Room driver/Media3/Resources/release) → `@android-platform-expert`; только web (JS-interop/WebGL/video/popstate) → `@wasmjs-expert`; рассинхрон платформ / потеря на сериализации / expect-actual → `@kmp-expert`. Баг на стыке двух слоёв → цепочка. Red-репро через `@test-expert` ДО фикса.

**Repro/harness — через реальный production-путь, не переписанную копию.** Диагностический скрипт/harness, реализующий пайплайн заново (свой парсинг, свой клиент, обходит сериализацию/делегат/сеть), доказывает только изолированную логику — баги живут на стыках, которые он обошёл, и harness остаётся «зелёным», пока прод падает. Harness ДОЛЖЕН вызывать те же production-функции и парсить тем же кодом. Прецедент 2026-06-08 (wasmJs face-embeddings): harness переписал pipeline в чистом JS → доказал, что математика верна, но прод падал на JSON-сериализации Float и WebGPU-делегате (harness их обошёл) → ~4 лишних итерации.

**Data-pipeline «фикс не помог» → инструментировать ОБЕ стороны каждой границы за ОДИН прогон.** Не добавлять diag по одному логу за round-trip (каждый = цикл с пользователем). Логировать с обеих сторон границы сразу (перед сериализацией + после парсинга; продюсер + консьюмер): первые N значений вектора/payload + сводку. Один прогон разводит «сторона A норм / сторона B мусор» (баг границы — сериализация/маппинг) vs «обе мусор» (источник — модель/входные данные) vs «обе норм / результат плохой» (алгоритм). Инкрементальное «добавил один лог → жду прогон → добавил ещё» сжигает итерации.

### Stuck-fix circuit breaker

**Триггер: 3 fail-loop'а подряд** (правка → запуск/билд/тест/проверка → провал). НЕ loop: диагностический лог, hook auto-fix форматирования, зелёный билд по дороге.

**STOP — не предлагать следующий патч.** Вслух в чате: (1) что менял в попытках 1/2/3; (2) симптом тот же или новый каждый раз (новый ⇒ фиксишь не тот слой); (3) на каком слое баг реально живёт (UI/ViewModel/repository/DI/контракт/state-машина); (4) какой один шаг слоем выше закрыл бы все три. Только потом правка — на слое из п.4. «Не вижу» → `AskUserQuestion` с 2-3 гипотезами слоёв.

**Запрещено как «4-я попытка»** (маркеры неверного слоя): `if (edgeCase) return` под симптом; `try-catch`/`runCatching` «чтобы не падало»; ручной reset state/повторный вызов «для надёжности»; второй источник правды (флаг `alreadyDone`); `delay()`/повтор через `LaunchedEffect` как фикс; удаление assertion'а/теста, ловящего баг. Список **не исчерпывающий**: ориентир на класс «заглушить проявление вместо фикса слоя» — любой аналогичный обход тоже под запретом. Sunk-cost: 3 потраченные попытки — сигнал сменить слой, не продолжать.

### Обратная связь на каждый user action

На **любое** действие пользователя — видимый отклик. Делать так:

```kotlin
// ✅ ввод пуст — сказать об этом, не выходить молча
if (stripped.isBlank()) { emitSideEffect(ShowSnackbar(HINT_EMPTY_INPUT)); return }
```

Отклик = SideEffect (snackbar/toast/dialog) с причиной + что делать дальше. Любая форма тихого раннего выхода (`if (x.isBlank()) return`, `return@launch`, пустой `when`-else) — юзер видит «зависание», это bug; правило про **любой** молчаливый skip, не только показанные формы. Silent fallback в data-слое (emptyList/null) — логировать `AppLogger.error` с причиной.

### Автономность — делать самому через инструменты, пользователь как резерв

Максимум работы — самому, через доступные инструменты (CLI, MCP, API). Что можно посмотреть/выполнить командой — смотреть/выполнять самому, не отправлять пользователя делать это руками.

- **Любой настроенный CLI** (`gcloud`, `gh`, `firebase`, `wrangler`, `aws`, `adb`, …) — читать логи/конфиги/статусы/ресурсы и выполнять операции самому. Нужно значение из Google Cloud / консоли, доступной через CLI, — `gcloud … describe/list/get` (или эквивалент) сам, а не «зайди в консоль и посмотри».
- Подключать пользователя только когда: **(а)** без него технически никак — его руки/физический доступ/секрет/OAuth/интерактивный логин/решение, которого нет в данных; либо **(б)** его участие реально экономит много времени (быстрее скажет, чем агент найдёт). По умолчанию — агент делает всё сам.
- Это вход в блок «👉 Сделай руками» ниже: он — **только** для случая (а). Прежде чем просить — проверить, не закроет ли задачу CLI/инструмент без пользователя.

### Ручные действия пользователя — инструкция внизу финального ответа

**Триггер:** задача упирается в действия, которые может сделать только пользователь руками — завести env-переменную/secret, прикрепить/вставить API-ключ или токен, зайти на сайт/в консоль (Firebase, Google Cloud, Play Console, Cloudflare, RevenueCat, GA4…) и что-то создать/включить/подтвердить, выдать доступ, пройти OAuth/логин.

**Правило:** в САМОМ НИЗУ последнего сообщения — отдельный блок `## 👉 Сделай руками` (всегда финальный блок, после итогов задачи):
- Нумерованные шаги, максимально простые: 1 шаг = 1 действие, без жаргона; имена и значения (переменная, ключ, раздел, scope) — точные, копируемые as-is.
- Прямые ссылки на конкретные страницы (нужный раздел консоли/настроек, не главная) — если ссылки существуют.
- Ручные шаги, всплывшие у субагентов по ходу задачи, главный собирает в этот же единый блок — не разбрасывать по тексту.

Нарушение: инструкция в середине ответа, только в промежуточном сообщении, или пересказ «нужно настроить переменные» без конкретных шагов — пользователь читает конец последнего сообщения.

### Definition of Done (gate перед завершением)

**`/end-session` — императив для Standard+** (или Trivial×Medium+ с COMPLETE): запустить ДО финального ответа. Без него gate НЕ пройден. Trivial×Low — без `/end-session`, ответ напрямую. Скилл прогоняет пункты stats/INDEX/deferred.

1. **Validation** — билд/тесты прошли.
2. **Impact scan** — все зависимости обновлены.
3. **Self-check** против FAILURE.
4. **`@doc-writer` COMPLETE** (Impact≥Medium / условие документирования) — `run_in_background: true`, в промпте `Шагов: N` + готовый `git diff --name-only`.
5. **INIT пропущен?** — нет активного документа для подпавшей задачи → в ответе строка `⚠️ INIT-стаб не создан`.
6. **Stats** — `STATS_ROW:` от COMPLETE → в `~/.claude/stats/doc-writer.md`.
7. **Solutions INDEX** — `INDEX_ROW:` → в `docs/solutions/INDEX.md` (создать с шапкой если нет).
8. **Improvements log** — задача меняла `~/.claude/{CLAUDE.md,agents,skills,settings.json}` → improvement-файл создан + строка в README index.
9. **Deferred work** — отложенный scope оформлен (`docs/todos/` + memory `## Deferred Work`); в ответе перечислить deferred-пункты. Оставшийся `// TODO:` — нарушение.

**Подсчёт `Шагов: N`** (для COMPLETE): шаг = одна попытка двинуть GOAL, ограниченная валидацией (build/test pass) или обнаружением неверного подхода. Edit→PASS=1; Edit→FAIL→fix→PASS=2; делегация (любое число правок внутри)=1; неверный подход переписан=+1. **НЕ шаг:** `/commit`, `/end-session`, скауты, doc-writer, git status/diff/log, чтение, отчёты, Task*.

### Отчёт Complex

`docs/reports/<slug>-<date>.md`: название/дата/описание; исследовано и реализовано; результаты validation; проблемы и retry; статус `Done`/`Partially Done`.

## Документирование нетривиальных решений

| Сложность × Impact | INIT | UPDATE | COMPLETE |
|---|---|---|---|
| Trivial × Low | — | — | — |
| Trivial × Medium | — | — | опц. (спец-условия) |
| Trivial × High | — | — | ✅ фон |
| Standard × Medium/High | стаб пишет главный | ✅ фон | ✅ фон |
| Complex × любой | ✅ doc-writer-субагент | ✅ фон | ✅ безусловно, фон |

**COMPLETE обязателен** (даже Trivial×Low/Medium) если: 2+ итерации / ошибка в начальном плане; 5+ файлов / новый модуль / миграция; recurring bug / app-wide perf fix; инфраструктурный фикс с нетривиальной причиной (CSP, Firebase config, DI, регионы, serialization).

**INIT (Standard) — стаб пишет главный сам** одним `Write` в `docs/active/<slug>-<date>.md` (не субагент — cold-start ради frontmatter): шапка (Статус, Дата, Start SHA, Тип, Сложность, Impact, Затронутые модули) + `## Цель` + `## Технический план` + пустые `## Лог итераций` / `## Выводы` / `## Предложения по улучшению агентов`. INIT-**субагент** — только для Complex.

**UPDATE и COMPLETE — всегда `run_in_background: true`** (не блокировать диалог). UPDATE — сразу после каждого специалиста (иначе документ застывает на INIT). **COMPLETE-слим:** главный передаёт готовый `git diff --name-only <START_SHA>..HEAD` + 3-5 строк key-findings; doc-writer НЕ делает git-археологию сам (один `git diff --stat` для `files_edited` ок). COMPLETE возвращает `STATS_ROW:`/`INDEX_ROW:`.

**Жизненный цикл статуса и архивация (`docs/active` → `docs/completed`).** Статус в шапке активного документа — ровно один из словаря: `In Progress` | `Done` | `Deferred` | `Planned` | `Partially Done`. Формат строго `**Статус:** <значение>` — без ведущего дефиса-буллета, без эмодзи (`✅`/`✓`) и хвостов в самой строке (детали — в `## Выводы`); иначе SessionStart-дайджест парсит мимо шапки и подхватывает промежуточный `**Статус:**` из тела. Завершённая задача = `**Статус:** Done` + **перенос файла в `docs/completed/`** (делает doc-writer COMPLETE Шаг 7; `docs/active/` остаётся только для незавершённого). Промежуточные отметки специалистов в `## Лог итераций` НЕ писать через `**Статус:**` — использовать `**Итог итерации:**`.

**Fallback без `/end-session` / без doc-writer.** Задача фактически завершена, но COMPLETE/end-session не запускались (Trivial, сессия оборвалась, классификация сменилась на ходу) — **главный сам** ставит `**Статус:** Done` в шапке и перемещает документ в `docs/completed/`. Не оставлять завершённое в `docs/active/` как `In Progress`. Прецедент 2026-06-11: 149 готовых задач висели In Progress в `docs/active/` (154 файла всего), дайджест показывал ложный фронт работ; причина — COMPLETE менял статус, но не архивировал, а часто и не вызывался.

**Hard scope guard в конце каждого промпта `@doc-writer`** (полные лимиты — `~/.claude/agents/doc-writer.md`):
```
ЗАПРЕЩЕНО: git add/commit/push; ./gradlew build/assemble/compile/test;
Edit/Write вне docs/, project memory, ~/.claude/stats/ (НЕ трогать *.kt/*.kts/*.gradle*/*.xml/*.json).
Требует выйти — STATUS: REJECTED.
```

## Chainable specialist briefs (recovery при сбое субагента)

Брифы специалистов **должны включать actionable patches** (точные пути + old_string/new_string или псевдо-код), не только high-level описание — для двух сценариев: (1) следующий в цепочке получает готовый контракт; (2) агент упал (timeout/content filter/API error) → главный применяет scope сам по патчам из брифа, не перезапуская цепочку (cold cache + риск повторной блокировки).

Включать (особенно для последнего в цепочке): точные пути; блоки до/после (`Edit old_string/new_string`); импорты добавить/удалить; KDoc для новых публичных API; **verification step** (1 строка — как проверить).
