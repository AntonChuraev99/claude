# Jira Task Writer

Преобразует любой входной контекст в чистое продуктовое описание Jira-задачи и сохраняет его как HTML-файл.

## Принципы

**Продуктовый уровень, не технический.** Описание читают PM, дизайнер, QA и разработчики обеих платформ. Ни слова о конкретных классах, файлах, паттернах, библиотеках или платформах (Android / iOS / Kotlin / Swift / Jetpack / UIKit).

**Платформо-нейтральный язык.** Задача покрывает обе платформы. Acceptance Criteria описывают поведение, которое QA может проверить на любой платформе, не зная стека.

**Язык вывода** = язык входа. Русский вход → русский вывод.

**Лаконичность — только полезный текст, без воды.** Каждое предложение несёт факт, который читатель не выведет сам. Контекст — 1 абзац (2-й — только если без него непонятно). Не повторяй одно и то же в разных секциях. Если фразу можно убрать без потери смысла — убери. Цель: задачу читают за 30 секунд, не продираясь сквозь воду.

**Нет технических деталей нигде в задаче.** Technical Notes, описание архитектуры, API-контракты, имена классов / методов / файлов, Firebase Function names, Remote Config ключи, Firestore пути, SDK-поля, флаги — **никогда** не попадают в Jira-задачу: ни в Story, ни в AC, ни в Флоу, ни в Аналитику, ни в любую другую секцию. Реализация — забота разработчика, она прочитает по коду. Из входа извлекай только продуктовое поведение.

Исключение для секции «Аналитика» — имена аналитических событий и параметров (это продуктовый контракт, не код): они идут в `<code>` как есть.

**Идентификаторы из кода не переносить.** Имена экранов, классов, функций, ключей конфигурации — переводить в продуктовый язык:
- ❌ "Открывается экран UploadClothesAttention"
- ✅ "Открывается экран-инструкция по загрузке"
- ❌ "URL берётся из commonSettings.outfitUploadInstruction.imageURL"
- ✅ "Изображение-пример загружается из конфигурации"

Если возник соблазн добавить раздел «Техническая реализация» / «Implementation notes» / «Architecture» — **это сигнал, что в задачу проникает код**. Такой секции в шаблоне нет намеренно. Если без неё пункт нельзя выразить — пункт надо убрать совсем (см. правила Acceptance Criteria ниже).

## Процесс

1. Прочитай входной контекст: черновик, техническое описание, Figma-ссылка, набор требований, их комбинация.
2. Если не хватает информации для AC или флоу — задай уточняющие вопросы **через `AskUserQuestion`** (2–4 готовых варианта на вопрос, не текстом в чат). Не додумывай то, что влияет на поведение продукта.
3. Сформируй HTML-файл по шаблону ниже и сохрани в `docs/plans/YYYY-MM-DD-<slug>-jira.html`.
4. Открой файл в браузере:
   - Windows: `start docs/plans/YYYY-MM-DD-<slug>-jira.html`
   - Mac: `open docs/plans/YYYY-MM-DD-<slug>-jira.html`
5. Скажи пользователю: открой файл в браузере → `Ctrl+A` → `Ctrl+C` → вставь в Jira.

## Почему HTML, а не Markdown

Jira's new editor не рендерит Markdown при вставке — всё превращается в один сплошной текст. HTML с семантическими тегами (`<h1>`, `<h2>`, `<ul>`, `<li>`, `<code>`, `<table>`) сохраняет структуру. CSS в `<style>` нужен только для красивого отображения в браузере — Jira его стрипает, это нормально.

Важно: `<h3>` в Jira рендерится слабо (выглядит как жирный текст). Всегда используй `<h1>` для названия задачи и `<h2>` для секций.

## Правила для каждой секции

### Заголовок
Краткий, продуктовый, по делу — что меняется для пользователя, не как сделано. Без имён классов / экранов / тех-жаргона.
- ❌ "Фикс DiscountHeader при нулевом discountPercent"
- ✅ "Колесо подарков: корректные тексты при нулевой скидке"

### Тип: Story или Проблема (баг)
Определи по входу — **что-то работает не так** (баг) или **новое/улучшение** (фича):
- **Фича / улучшение** → секция **Story** («Как …, я хочу …, чтобы …»).
- **Баг** → вместо Story секция **Проблема**: `Шаги → Ожидаемо → Сейчас`. Story-обёртку для бага не натягивай («Как пользователь, я хочу, чтобы не было бага» — плохо).
  - Если из входа известно — добавь **Повторяемость** (всегда / периодически / один раз) и **Условия** на продуктовом уровне (тип устройства, версия ОС/приложения, сеть, free/Premium-аккаунт). НЕ тяни build-номера, стек-трейсы, логи — это код.
  - Есть скриншот/запись — впиши как доказательство; если их нет во входе, напомни пользователю приложить (в финальном сообщении, не в самой задаче).
  - НЕ добавляй Severity / Priority — это поля триажа в Jira, не текст задачи.

### Story
- Тип пользователя: free-пользователь, Premium-пользователь, неавторизованный, новый пользователь. Не "разработчик".
- Действие — что хочет сделать, не как реализовано.
- Ценность — реальная польза, не техническая цель.

### Acceptance Criteria

AC — это **только то, что нельзя вывести из Флоу**. Если поведение уже описано в секции Дизайн — в AC его нет.

**Если без технических деталей пункт пустой — выкидывай его совсем.**
Если для выражения сути нужны SDK-поля, ключи entitlement-ов, имена методов или Firebase Function names — это деталь реализации. Без них пункт становится либо тривиальным («работает корректно»), либо непонятным QA. В обоих случаях — он не AC. Эту информацию разработчик найдёт в коде.

Пример с `isSandbox`:
- ❌ AC: "Флаг берётся из entitlement подписки; для генераций — из entitlement генераций" — детали реализации в AC
- ❌ AC: "Значение `isSandbox` определяется динамически; не захардкожено" — без технических деталей пункт ничего не говорит QA
- ✅ Решение: пункт удаляется из AC целиком. Источник `isSandbox` — деталь реализации, место которой в коде, не в Jira.

**Хорошие AC** — правила, которые QA не выведет из флоу сам:
- ✅ "В строке аутфитов шторки порядок: аутфиты каталога → загруженные пользователем → карточка '+'" — правило порядка UI
- ✅ "Premium-пользователю показана кнопка загрузки фото; пользователю без подписки — кнопка перехода к оформлению подписки" — Premium-гейт
- ✅ "Экран-инструкция отображает пример изображения, загружаемого из конфигурации" — источник данных
- ✅ "На экране профиля, после блока 'Your faces', появляется секция 'My outfits'; карточка '+' расположена в конце" — порядок/состав UI

**Плохие AC** (удалить):
- ❌ "Нажатие '+' открывает экран-инструкцию" — дублирует флоу
- ❌ "После загрузки аутфит появляется в списке" — очевидное следствие
- ❌ "Экран-инструкция отображает заголовок и описание" — деталь верстки
- ❌ "При ошибке загрузки показывается сообщение об ошибке" — нет специфики: какое сообщение, в каком виде?

Цель: 3–5 пунктов.

Для UI-задач проверь **состояния экрана** (loading / пусто / ошибка): если поведение в них значимо и QA не выведет из флоу — добавь пункт. Не выдумывай состояния, которых в задаче нет.

**Каждый AC объективно проверяем** (есть однозначный pass/fail). Бан размытых формулировок без критерия: «работает хорошо», «быстро», «удобно», «корректно». Не проверить однозначно → переформулируй в измеримое или выкинь.

**Сложный AC с условием** (исход зависит от состояния / роли / предусловия, и bullet это сплющивает) — допустим формат `Дано → Когда → Тогда` для такого пункта. НЕ делай это стилем по умолчанию: для простых правил bullet короче. GWT — только когда предусловие критично и в bullet выпадает.

### Флоу (секция Дизайн)

Лаконичная стрелочная нотация. Детали — в Figma. Не нумеруй шаги, не описывай очевидные переходы. Не добавляй секцию Edge cases — это дублирует AC.

**✅ Хорошо:**
```
Загрузка: шторка генерации / профиль → "+" → экран-инструкция → фотопикер → загрузка → возврат назад
Удаление (вариант 1): профиль → аутфит → полноэкранный просмотр → "Remove Outfit" → аутфит удалён
Удаление (вариант 2): профиль → долгое нажатие на карточку → шторка действий → "Remove Outfit" → аутфит удалён
```

**❌ Плохо** — лишние детали, нумерация, очевидные переходы:
```
Загрузка:
1. Пользователь нажимает кнопку "+" в строке аутфитов в шторке генерации или на экране профиля
2. Открывается экран-инструкция по загрузке одежды
3. Пользователь нажимает кнопку "Загрузить фото"
4. Открывается системный фотопикер
...

Edge cases:
- При ошибке загрузки показывается сообщение
- Если пользователь не Premium — показывается кнопка подписки
```

### Тексты (когда меняется копирайт)

Если задача **меняет видимые пользователю тексты** (формулировки на экране, замена строк, разный текст при разных состояниях) — добавь секцию «Тексты» с таблицей сравнения. Точные строки на языке приложения — это продуктовый контент, **не** код: их нужно показывать дословно, QA и PM сверяют буквально.

Форму таблицы выбирай по природе изменения:
- **Замена (было → стало):** колонки `Место | Было | Стало`.
- **Текст зависит от состояния/условия:** колонки `Место | <состояние A> | <состояние B>`. Пример: `Место | При скидке >0% | При скидке 0%`. Если одно из состояний не менялось — всё равно покажи его (колонка-референс), чтобы было видно контраст.

Правила:
- **Одна таблица**, все затронутые места — строками (заголовок, кнопка, шторка, баннер, …). Не дроби на несколько таблиц по местам.
- Различие выделяй внутри ячейки (`<strong>`) — например процент или ключевое слово, которое и есть суть правки.
- Тексты пиши **как в приложении** (на их языке — английском и т.п.), не переводи.
- **Когда НЕ нужна:** копирайт не трогается (новый функционал без правки существующих строк) — секцию пропусти.

Пример (реальная задача — обработка нулевой скидки на колесе подарков):

| Место | При скидке >0% (пример 30%) | При скидке 0% |
|---|---|---|
| Заголовок колеса | Spin to get your **30%** discount, forever! | Spin to claim your exclusive gift! |
| Баннер таймера | **30%** + заголовок оффера | только заголовок оффера |

### Аналитика
- Имя события в `<code>`.
- Триггер — действие пользователя, видимое глазу. Не "ViewModel.init".
- Параметры: если параметр принимает фиксированный набор значений — указывай **точные строки** с пояснением контекста:
  - ❌ `location` — точка входа (шторка генерации или профиль)
  - ✅ `location` — точка входа: `"photoshoot"` (шторка генерации) или `"profile"` (экран профиля)
- Если аналитика не указана во входе — пропусти секцию.

### Вне скоупа
- Только если граница неочевидна и важна.
- Пустую секцию не добавлять.
