# Auto-Commit Rules — Step 5.1.1

Этот файл — extract из Step 5.1.1 `end-session/SKILL.md`. Точные guard-условия для автоматического вызова `/commit` без участия пользователя.

## Принцип

`/commit` — это разрешённый и единственно правильный путь создавать коммиты в этом окружении (см. глобальный `CLAUDE.md`). Если все остальные проверки gate прошли, ручное вмешательство пользователя для финального шага создаёт лишнюю задержку — он уже мысленно завершил задачу. Скилл должен закрывать gate целиком.

## Условия активации (ВСЕ одновременно)

Auto-commit запускается, только если **все 5 условий** выполнены:

### 1. Gate без блокеров

В пунктах 2.1–2.9, Step 3, Step 4 — только `✅` или `⚠️` (warnings допустимы, **ни одного `❌`** блокера).

### 2. Есть несохранённое состояние

В 5.1 коммит не сделан (или сделан, но после него появились новые правки) и `git status --porcelain` непустой.

### 3. Пользователь явно НЕ запретил

Пользователь не сказал в этой сессии:
- «без коммита»,
- «коммит сделаю руками»,
- «отложу коммит»,
- «не коммить»,
- любая явная формулировка отказа от коммита.

### 4. В diff нет подозрительных файлов

**Black list (auto-commit БЛОКИРУЕТСЯ при наличии):**

- `.env*` (любые env-файлы)
- `*.key`, `*.pem` (приватные ключи)
- `id_rsa*` (SSH-ключи)
- `*credentials*`, `*secret*` (любые файлы с такими подстроками)
- Бинарники > 1 МБ
- Файлы вне ожидаемого scope задачи (например, modified `~/.zshrc` при задаче по Android-коду)
- Файлы в системных директориях, которые не должны меняться (`/etc/`, `/usr/`, и т.п.)

Проверка:
```bash
git diff --name-only --cached | grep -iE '\.env|\.key|\.pem|id_rsa|credentials|secret'
git diff --name-only HEAD     | grep -iE '\.env|\.key|\.pem|id_rsa|credentials|secret'
# Размер бинарников
git status --porcelain | awk '/^[AM?]/ {print $2}' | xargs -I{} ls -la {} 2>/dev/null | awk '$5 > 1048576'
```

### 5. 2.9 без unbacked TODO

Если есть `// TODO:` / `// FIXME:` / `// STOPSHIP:` без `// Pending: docs/todos/...` anchor'а — auto-commit НЕ запускается до явного решения пользователя (см. `references/deferred-work-scan.md`, Шаг 3).

## Алгоритм запуска

Когда все 5 условий выполнены:

1. **Вызвать** `Skill(skill="commit")`. `/commit` сам соберёт сообщение по Conventional Commits, поставит файлы и сделает коммит.
2. **Дождаться** завершения `/commit`.
3. **Перепроверить** `git status --porcelain`:

| Результат | Действие |
|---|---|
| Чисто | Отметить `✅ commit auto-created`, в финальный отчёт включить SHA нового коммита, итог `✅ READY` |
| Не чисто (остались untracked, /commit отказался, упал pre-commit hook) | Отметить `❌ auto-commit failed`, в отчёт добавить полный stderr `/commit`, **не** заявлять задачу завершённой, передать управление пользователю |

## Блокирующие сценарии (auto-commit НЕ запускается)

Даже если `git status --porcelain` непустой, auto-commit пропускается в этих случаях:

### Любой `❌` блокер выше по gate

Сначала закрыть блокеры, коммит — после. В отчёте: `5.1 Commit ❌ blocked by gate failures`.

### Пользователь явно отложил коммит

В отчёте: `5.1 ⚠️ commit deferred (user request)`, идти к 5.2 с вердиктом `⚠️ READY WITH WARNINGS`.

### Подозрительные файлы в diff

НЕ автокоммитить, спросить пользователя что с ними делать через `AskUserQuestion`. Опции:

| Опция | Действие |
|---|---|
| **Исключить файл из коммита и продолжить** | `git restore --staged <file>` или `git reset HEAD <file>`, повторить sanity check, продолжить auto-commit |
| **Просмотреть содержимое перед решением** | Показать `git diff <file>`, затем повторить вопрос |
| **Закоммитить как есть (явный override)** | Пометка `⚠️ commit blocked override: suspicious files`, продолжить auto-commit, в commit body упомянуть override причину |
| **Отложить коммит** | `⚠️ commit deferred (suspicious files)`, идти к 5.2 |

Зафиксировать в отчёте: `⚠️ commit blocked: suspicious files (<list>)`.

### Diff отражает заведомо промежуточное состояние

Признаки промежуточного состояния:
- Задача с retry / частичным успехом, статус `Partially Done` в отчёте.
- В диффе много вспомогательных diagnostic-логов (`println(`, `console.log(`, `Logger.d(`), которые планировалось удалить.
- Diff содержит файлы из разных задач, ещё не сепарированных.

НЕ автокоммитить, спросить через `AskUserQuestion`: «Корректный ли это снапшот для коммита?» с опциями:
- **Yes, commit as is** → продолжить auto-commit.
- **No, let me clean up first** → передать управление пользователю, не коммитить.
- **Show me what's staged** → показать `git diff --cached`, затем повторить вопрос.

## Почему такой жёсткий гард-список

Каждое из условий — следствие реального инцидента в истории проекта:

- **Условие 4 (suspicious files)** — секреты в публичных репо стоят дороже любой задержки. Auto-commit без проверки = потенциальная утечка.
- **Условие 5 (unbacked TODO)** — прецедент 2026-05-12 (см. `references/deferred-work-scan.md`).
- **Условие 3 (explicit user refusal)** — уважение к явному выбору пользователя; нарушение = доверие к скиллу падает.
- **Условие 1 (no blockers)** — auto-commit пытается закрыть gate, но если gate уже сломан, коммит не помогает.
- **Условие 2 (есть что коммитить)** — без diff auto-commit не имеет смысла; пустой коммит — мусор.

## Когда auto-commit особенно полезен

Auto-commit максимально окупается при типичных циклах:
- Standard задача с одним специалистом + COMPLETE doc-writer.
- Gate полностью зелёный.
- Пользователь сказал «готово» или «заканчиваем».
- Никаких подозрительных файлов в diff.

В таких случаях скилл закрывает всё одной транзакцией без участия пользователя.
